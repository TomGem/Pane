How Claude Code Works

For the non-developers in the room

Describe → Read → Write → Review → Commit

You describe what you want in plain English.

Claude reads the existing code, writes the changes, and explains what it did.

You review, ask for adjustments, and approve.

The CLAUDE.md

The Most Important Artifact


264 lines of structured documentation.

Updated with every major feature. It describes:

  • Architecture & data flow
  • Every API route and its behavior
  • Database schema and conventions
  • UI patterns and component structure
  • Environment variables
  • Coding conventions & rules

It serves two purposes:


For Claude — Context that persists across sessions. Without it, each conversation starts from zero.

For humans — Living documentation. Always accurate because the AI that writes the code also writes the docs.

The Permission System

Trust Boundaries in AI Collaboration


Claude Code uses settings.local.json to define what it's allowed to do:


{
  "permissions": {
    "allow": [
      "Bash(pnpm dev)",
      "Bash(pnpm build)",
      "Bash(pnpm check)",
      "Bash(docker compose *)",
      "Bash(git *)"
    ]
  }
}

  • Every tool call can be approved, denied, or pre-authorized
  • File edits, shell commands, and git operations are separate permissions
  • You decide how much autonomy to grant

Security-conscious collaboration: the AI proposes, the human disposes.

What Went Well

CLAUDE.md discipline — Keeping the project document updated paid off exponentially. Each session started productive because Claude had full context.

Full commitment to the approach — Zero lines written manually. Every change went through the AI. This forced consistent patterns and thorough documentation.

Iterative cycles — Small, reviewable changes. Not "generate the whole app" — but hundreds of focused conversations.

Deliberate stress testing — The multi-user pivot was intentionally ambitious. It proved the approach scales to major architectural changes.

Consistent quality at any hour — Claude writes the same quality code at 2 AM as at 2 PM. The human's review quality, however, varies.

What Could Be Improved

No automated tests — Only type-checking via pnpm check. A real project needs test coverage. Claude can write tests — I just never asked.

No cross-session memory — Each Claude Code session starts fresh. The CLAUDE.md helps, but Claude doesn't remember yesterday's debugging session.

Oversized commits — The multi-user pivot was one massive commit. Smaller, reviewable steps would have been safer.

No second reviewer — Solo project, sole reviewer. Fresh eyes would have caught things both the AI and I missed.

Late-night decision-making — Most commits happened between 10 PM and 3 AM. Good for focus, questionable for judgment.

Honest Observations

Bug fix chains are normal — A feature often works on the first try. But when it doesn't, fixing one thing can break another. This is normal software development — AI doesn't eliminate it.

Session boundaries matter — Long sessions accumulate context and work well. Starting a new session for "one quick fix" often takes longer than expected because Claude needs to re-orient.

Security needs explicit attention — Claude doesn't automatically think about SSRF, XSS, or enumeration attacks. You have to ask. But when asked, it handles them well.

AI excels at patterns — Once you establish a pattern (API route structure, component layout, error handling), Claude replicates it perfectly. The human drives architecture — the AI executes consistently.

Lessons Learned


1. Invest in your CLAUDE.md
It's the single highest-leverage thing you can do. Every minute spent documenting saves ten later.
2. Review everything
AI-generated code is not automatically correct. Read every diff. Understand every change.
3. Work iteratively
Small changes, frequent commits. Don't ask for "the whole feature" — build it step by step.
4. Test deliberately
If you don't have automated tests, test manually after every change. AI won't catch its own regressions.
5. Be explicit about security
Don't assume the AI thinks about edge cases. Ask specifically about attack vectors.
6. Establish patterns early
Good conventions compound. A consistent API structure means every new route is predictable.
7. Know what you're building
AI replaces typing, not thinking. You still need to understand what the code does and why.

Could You Do This?

If you're a developer

Yes. You'll be more productive than I was. Your existing knowledge lets you review faster, catch issues earlier, and guide the AI more precisely.

The learning curve is in how to communicate with the AI, not in the technology itself.

If you're new to this

Yes — and this might be the best time to start. You won't understand everything right away, and that's completely fine. Neither did I.

Every prompt teaches you something. Every error message is a lesson. You'll read things you've never seen before, look them up, and suddenly know more than you did five minutes ago.

Start the first prompt. Discover new and unknown things. Read more. Learn more. Prompt on.

Where to Start

A practical path for beginners

1
Pick one small thing you actually want to build.
A personal website. A to-do list. A calculator. Something simple that you'd use. Motivation beats ambition.
2
Install an AI coding tool and describe your idea in plain language.
Claude Code, Cursor, GitHub Copilot — pick any. Say what you want, not how to build it. "I want a page that shows a list of my favorite books."
3
Read what the AI gives you, even if you don't understand it yet.
You'll recognize words over time. Ask "explain this to me" — the AI is a patient teacher. Don't skip this step.
4
Run it. Break it. Fix it.
Paste the error back to the AI. This loop — try, fail, ask, learn — is how every developer started. The AI just makes the loop faster.
5
Grow one feature at a time.
Don't plan a big system. Add one thing, see it work, then add the next. Complexity grows naturally when you're ready.

The gap between "no idea" and "first working thing" has never been smaller.
